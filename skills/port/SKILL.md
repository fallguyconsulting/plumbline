# /plumbline:port

Write a complete Plumbline port plan for the current project to a file. The plan enumerates every violation across both checks and the passes needed to make `plumbline .` return clean. Output goes to a markdown file — ready to hand off to a planner or to execute directly.

For the full reference on each phase, see `docs/plumbline-porting-guide.md` in the plugin distribution.

## What this does

1. Runs `plumbline doctor` to assess installation state.
2. Runs `plumbline .` to count violations per check.
3. Runs `plumbline patterns` to cluster the violations by shape — this is what makes the `comment-hygiene` pass concretely actionable instead of "thousands of them, good luck."
4. Writes a phase-by-phase plan to `./plumbline-port-plan.md` (or the path passed as the second arg). The plan's goal is **complete**: executing it leaves the lint clean with both checks on. The budget ratchet, which is a strategy for projects that won't do the full sweep at once, is referenced as a fallback in the porting guide, not in the plan itself.

The plan skips the **Adopt** pass when doctor reports `healthy`. It skips each check's pass when that check's count is zero. The total passes always include **Maintain** as the steady-state description.

## Run

```bash
plumbline_bin="${CLAUDE_PLUGIN_ROOT%/}/bin/plumbline"
guide_path="${CLAUDE_PLUGIN_ROOT%/}/docs/plumbline-porting-guide.md"
target="${1:-.}"
output="${2:-./plumbline-port-plan.md}"

abs_target=$(cd "$target" && pwd)
project=$(basename "$abs_target")

doctor_out=$(node "$plumbline_bin" doctor "$target" 2>&1 || true)
doctor_healthy=false
if echo "$doctor_out" | grep -q "plumbline doctor: healthy"; then
  doctor_healthy=true
fi

audit_out=$(node "$plumbline_bin" "$target" 2>&1 || true)
patterns_out=$(node "$plumbline_bin" patterns "$target" 2>&1 || true)

hygiene_count=$(echo "$audit_out" | grep -c "plumbline/comment-hygiene" || true)
citation_count=$(echo "$audit_out" | grep -c "plumbline/citation-unresolved" || true)
total=$((hygiene_count + citation_count))

{
  echo "# Plumbline port plan — $project"
  echo ""
  echo "Generated by \`/plumbline:port\`."
  echo ""
  echo "**Goal**: execute this plan and \`plumbline .\` returns clean with both checks (\`comment_hygiene\`, \`citation_resolution\`) enabled."
  echo ""
  echo "Reference: \`$guide_path\`"
  echo ""
  echo "## Current state"
  echo ""
  echo '```'
  echo "$doctor_out"
  echo '```'
  echo ""
  echo "## Backlog"
  echo ""
  echo "| Check | Count |"
  echo "| --- | --- |"
  echo "| comment-hygiene | $hygiene_count |"
  echo "| citation-unresolved | $citation_count |"
  echo "| **Total** | **$total** |"
  echo ""

  if [ "$total" -eq 0 ] && [ "$doctor_healthy" = "true" ]; then
    echo "## Result"
    echo ""
    echo "Already Plumbline-clean. Nothing to do."
    echo ""
  else
    echo "## Plan"
    echo ""

    pass=1

    if [ "$doctor_healthy" != "true" ]; then
      cat <<EOF
### Pass $pass — Adopt

- Install the plumbline plugin in this project
- Run \`/plumbline:affirm\`; commit \`.claude/rules/plumbline-cheatsheet.md\`
- Run \`/plumbline:starter .\`; review the proposed config; save as \`.plumbline.json\`; commit
- Verify: \`/plumbline:doctor\` reports \`healthy\`

EOF
      pass=$((pass + 1))
    fi

    if [ "$citation_count" -gt 0 ]; then
      cat <<EOF
### Pass $pass — \`citation-unresolved\` ($citation_count violations)

Citation tags are declared in \`.plumbline.json\`'s \`citations\` array. Each violation is a slug that does not resolve.

- Run \`/plumbline:patterns\` to cluster by tag (\`tag:@concept:\`, \`tag:@story:\`, ...).
- For each tag cluster: triage each slug — fix typos, rename to match an existing artifact, create the missing artifact, or remove the citation if the link is no longer load-bearing.

Verify: \`plumbline .\` reports zero \`citation-unresolved\` violations. Commit.

EOF
      pass=$((pass + 1))
    fi

    if [ "$hygiene_count" -gt 0 ]; then
      cat <<EOF
### Pass $pass — \`comment-hygiene\` ($hygiene_count violations)

Under plumbline's strict rule, comments are not permitted in source files except machine directives (license headers, lint suppressions, build tags), configured citation tags, and JSDoc/GoDoc comments in files carrying the \`@plumbline:allow-docstrings\` opt-in marker. Every violation listed is one of: an obvious deletion (residue, restatement, TODO, divider), a load-bearing constraint that should become an assertion / test / type / name, or a docstring on a public-API surface that warrants the opt-in marker.

Cluster shapes (from \`plumbline patterns\`):

\`\`\`
EOF
      echo "$patterns_out"
      cat <<'EOF'
```

For each cluster:

- `disallowed-prose` / `todo-marker` / `divider` / `commented-out-code` — sample 3-5, confirm the shape, bulk delete.
- `doc-residue` — the comments are JSDoc/GoDoc-shaped but the file is not opted in. If the file is a public-API surface, add `// @plumbline:allow-docstrings` near the top (with `#` for hash-comment languages). Otherwise delete.
- `license-fragment` — comments mention a license but don't match the machine-directive patterns. Reformat so the license header opens with `SPDX-License-Identifier:` or `Copyright`, or delete the residue.
- For each comment that names a real constraint (a `must`/`always`/`requires` style invariant, a deliberate-choice guard), run `/plumbline:suggest` for per-violation proposals and convert to code (assertion with message, test that fails on violation, type that enforces, function/variable name that carries the intent).

Verify: `plumbline .` reports zero `comment-hygiene` violations. Commit.

EOF
      pass=$((pass + 1))
    fi

    cat <<EOF
### Pass $pass — Maintain (steady state)

- The \`PostToolUse\` hook blocks edits that introduce new violations
- Run \`/plumbline:ci <platform>\` and add the emitted workflow to CI

EOF

    cat <<'EOF'
## Notes for the executor

- Each pass must leave the lint clean for the checks it addresses before the next pass begins.
- The default action for a comment-hygiene violation is **delete**. Conversion to code (assertion / test / type / name) is the right move only when the comment names a real constraint that some downstream agent could break by "simplifying."
- If the project explicitly decides not to do the full `comment-hygiene` sweep at once, the **Ratchet fallback** (`plumbline budget save` + the CI budget-check step) captures a baseline and prevents regression while the backlog burns down opportunistically. The fallback path is documented in the porting guide referenced at the top — it is not part of this plan, because this plan's goal is the clean-state contract.

EOF
  fi
} > "$output"

echo "Wrote port plan: $output"
echo "Backlog: $total total violations."
```

## After the script runs

The plan is at the path printed by the script (default `./plumbline-port-plan.md`).

Before handing it off, check whether the project uses a plan/spec convention and offer to move or adapt the file accordingly:

- Look for directories like `.ok-planner/`, `.specs/`, `docs/plans/`, `plans/`, `proposals/`, `design/`
- Check the project's `CLAUDE.md` or `README.md` for a documented plan/spec workflow
- If found, ask the user whether to move the file into that convention (matching its naming — date prefix, sequence number, etc.) and whether to reshape headings if the convention expects a specific structure
- If none found, leave the file at the default path

If the project has a planning orchestrator (a `/brainstorm`, `/write-plan`, or similar), suggest invoking it with this file as the input.

The plan's goal — `plumbline .` returns clean with both checks enabled — is the binding contract for whatever executes it.
